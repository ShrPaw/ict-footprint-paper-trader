## THE PROJECT: ICT + Footprint Paper Trader

### What It Is
A regime-adaptive paper trading engine for crypto perpetuals (ETH, SOL, BTC, XRP).
Uses ICT (Inner Circle Trader) concepts on 1H candles combined with order flow footprint analysis.
Built in Node.js. Trades on Hyperliquid (live market data). Backtests on Binance (historical).
Paper trading only — NO real money at risk. Generates signals, tracks virtual PnL.

### The Core Idea
Different crypto assets behave differently in different market regimes. Instead of one strategy
for all assets in all conditions, the system:
1. Detects the current market regime (TRENDING_UP, TRENDING_DOWN, RANGING, VOL_EXPANSION, LOW_VOL)
2. Blocks trading in regimes where each asset historically loses money
3. Only trades when the regime favors the specific asset

## WHAT'S EXCELLENT

1. **Architecture is solid** — Per-bot independence (own engine, analyzers, feed). No shared state
   means no cross-contamination bugs. Clean separation of concerns.

2. **Per-asset regime blocking is the real edge** — Data-driven discovery baked into config.
   BTC/XRP block RANGING, SOL works best in RANGING. Configurable, testable, data-proven.

3. **Exit logic is well-designed** — Partial TP at 1.5x ATR (100% WR), trailing stop at 0.9x
   activation (91-100% WR), smart time_exit (only if loss > 0.5x ATR after 4h).

4. **Backtest engine is comprehensive** — By-regime, by-signal, by-exit, by-mode, by-asset
   breakdowns. CSV export. Equity curve. Sharpe calculation. Better than most retail tools.

5. **Real footprint from Hyperliquid** — Actual buy/sell delta from trade-level data, not
   estimated from OHLCV like most "order flow" tools.

6. **Telegram integration with rate-limiting queue** — Proper implementation with 350ms
   spacing between messages.

7. **Deployment-ready** — PM2 config, deploy.sh, one-command VPS setup.

## WHAT'S BAD / NEEDS FIXING

1. **RegimeDetector ADX is simplified** — Not using Wilder's smoothing. ADX values used for
   regime classification thresholds may be wrong, affecting ALL downstream logic.

2. **Footprint estimation from OHLCV is crude** — closePosition = buy/sell ratio. Barely
   better than random for signals weighted 60-75% of scoring.

3. **BTC/XRP sample sizes are thin** — 26 and 44 trades over 5 months. Need 100+ for
   statistical confidence.

4. **TRENDING_UP blocked in two places contradicting each other** — Line 48 says "re-enable"
   but line 156 blocks it again. Confusing.

5. **No state persistence** — Bot crash = lost trade history. Need JSON dump every minute.

6. **Dashboard has no auth** — Anyone who finds port 3500 can see trading activity.

7. **No walk-forward validation** — Single period (Jan-May 2025). Edge might be curve-fit.
   Need train 3mo, test 1mo, slide.

8. **Jul-Nov 2024 bull run loses** — PF 0.69 ETH, 0.96 SOL. System misses strong trends.

9. **ecosystem.config.cjs was broken** — Used `env_file` (invalid PM2 field). FIXED this session.

10. **Weekend/Scalping code still imported** — Dead weight. Should be removed or flagged.
